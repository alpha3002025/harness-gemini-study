### A 

Gemini CLI에서 **Explore** 에이전트와 가장 유사한 역할을 수행하는 것은 **`codebase_investigator`**입니다.

이 에이전트는 다음과 같은 이유로 해당 워크플로우에 적합합니다:
- **아키텍처 매핑 및 분석:** 코드베이스의 전반적인 구조와 시스템 간의 의존성을 파악하는 데 특화되어 있습니다.
- **설계 의도 파악:** 단순한 코드 읽기를 넘어, 프로젝트의 맥락과 설계 의도를 분석하여 보고서 형식으로 결과를 제공합니다.
- **문서와 코드의 연계:** `/docs/` 내의 설계 문서(PRD, ADR 등)와 실제 구현된 코드를 대조하며 조사하는 작업에 유용합니다.

또한, 보다 광범위한 조사나 반복적인 데이터 처리가 필요한 경우에는 **`generalist`** 에이전트를 보조적으로 활용할 수 있습니다.



## Q2
```markdown
### C. Step 설계

사용자가 구현 계획 작성을 지시하면 여러 step으로 나뉜 초안을 작성해 피드백을 요청한다.

설계 원칙:

1. **Scope 최소화** — 하나의 step에서 하나의 레이어 또는 모듈만 다룬다. 여러 모듈을 동시에 수정해야 하면 step을 쪼갠다.
2. **자기완결성** — 각 step 파일은 독립된 Claude 세션에서 실행된다. "이전 대화에서 논의한 바와 같이" 같은 외부 참조는 금지한다. 필요한 정보는 전부 파일 안에 적는다.
3. **사전 준비 강제** — 관련 문서 경로와 이전 step에서 생성/수정된 파일 경로를 명시한다. 세션이 코드를 읽고 맥락을 파악한 뒤 작업하도록 유도한다.
4. **시그니처 수준 지시** — 함수/클래스의 인터페이스만 제시하고 내부 구현은 에이전트 재량에 맡긴다. 단, 설계 의도에서 벗어나면 안 되는 핵심 규칙(멱등성, 보안, 데이터 무결성 등)은 반드시 명시한다.
5. **AC는 실행 가능한 커맨드** — "~가 동작해야 한다" 같은 추상적 서술이 아닌 `npm run build && npm test` 같은 실제 실행 가능한 검증 커맨드를 포함한다.
6. **주의사항은 구체적으로** — "조심해라" 대신 "X를 하지 마라. 이유: Y" 형식으로 적는다.
7. **네이밍** — step name은 kebab-case slug로, 해당 step의 핵심 모듈/작업을 한두 단어로 표현한다 (예: `project-setup`, `api-layer`, `auth-flow`).
```

여기에서 `**AC는 실행 가능한 커맨드**` 은 무슨 의미인가요? 'AC'가 무슨 단어의 약자인가요?<br/>
<br/>

바로 아래의 ### A 에 답변을 작성하세요. 질문을 수정하거나 삭제하지 마세요.

### A

**AC**는 **Acceptance Criteria**(수용 기준)의 약자입니다.

소프트웨어 개발에서 AC는 특정 기능이나 작업이 "완료(Done)"되었다고 판단하기 위해 충족해야 하는 구체적이고 명확한 조건들을 의미합니다. 

본 워크플로우에서 **"AC는 실행 가능한 커맨드"**로 작성해야 한다는 원칙은 다음과 같은 설계 의도를 가지고 있습니다:

1.  **검증의 객관성:** "~가 동작해야 한다"는 식의 주관적인 서술은 해석의 여지가 있습니다. 대신 `npm run build && npm test`와 같은 실제 명령어를 명시함으로써, 작업 결과가 성공인지 실패인지 기계적으로(객관적으로) 확인할 수 있게 합니다.
2.  **에이전트의 자기 완결성:** 작업을 수행하는 에이전트(Claude 세션 등)가 자신이 작성한 코드가 올바른지 스스로 검증하고 피드백을 루프를 돌릴 수 있는 수단을 제공합니다.
3.  **신뢰성 보장:** 설계자가 의도한 핵심 규칙이 실제로 지켜졌는지 "실행"을 통해 증명하게 함으로써 전체 프로젝트의 품질을 유지합니다.

요약하자면, **AC(Acceptance Criteria)**는 해당 Step의 **'합격 통지서'를 받기 위한 시험 문제**와 같으며, 이를 **자동화된 명령어** 형태로 제시하여 검증의 효율과 정확도를 높이는 것입니다.
